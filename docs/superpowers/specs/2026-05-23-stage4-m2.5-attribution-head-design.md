# Stage-4 Milestone 2.5 — Learned Attribution Head — Design Spec

> **Status:** design draft 2026-05-23. Implementation-ready.
> **Why this exists:** Stage-4 M2a closes G1/G2/G4 end-to-end on real
> ManiSkill and matches `babysteps_selective` on a per-seed basis. The
> StackCube G5-strict gap (latent −10 pp vs `oracle_factor_revision`)
> is **not** a latent-step defect — it is the rule-based
> `babysteps.failure.attribute_failure` misclassifying 5/50 evaluation
> seeds. `goal.md` §"Stage 4 / Data Dependencies" explicitly defines
> this gap as the learned attribution head's job, not M2a's:
>
> > The hand-coded `failure.py` rules become the **teacher** of the
> > learned Stage-4 components, not the thing they replace.
>
> Authority: `goal.md` §"Stage 4 / Data Dependencies" + §"Stage 4 /
> Success Criteria". Code source: `babysteps/failure.py`,
> `babysteps/schemas.py`, `babysteps/stage4/latent_policy.py`,
> `babysteps/policies.py`.

---

## 1. Scope & Non-Goals

**In scope:**

- A `wrong_factor` classifier
  `AttributionHead: FailurePacket × Intent → wrong_factor ∈ INTENT_FIELDS`
  that replaces the rule-based `attribute_failure` for the
  `latent_revision` policy.
- Sim-free, CPU-only training and eval. Reuses existing
  `datasets/stage0_baselines/*/{PushCube-v1,StackCube-v1}/samples.jsonl`
  and `datasets/stage4/varied_intent/*` for supervision pairs.
- Trained-pack save/load. Extends `LatentPack` with an optional
  `attribution_head` field.
- Drop-in `latent_revision_factory` override:
  `ctx.attribution.wrong_factor ← learned head.predict(fp, intent)`
  before slot-local ReviseHead runs.

**Non-goals:**

- Change Stage-0 baseline semantics (rule-based attribution stays the
  canonical baseline; learned attribution is *new* code, not a
  replacement of `babysteps.failure.attribute_failure`).
- Change ReviseHead, slot_decode, IntentHead, or any M2a surface.
- Change Stage-0 episode loop (`episode.run_episode`) beyond what is
  already extended for M2a.
- Cross-task generalization (per-task heads, like per-task IntentHeads
  in M2a).
- New simulator probes / new privileged fields. Use only the existing
  FailurePacket fields + the chosen Intent's existing fields.

## 2. Why the Rule Misses 5/50 StackCube Seeds

From `datasets/stage0_baselines/oracle_factor_revision/StackCube-v1/samples.jsonl`
+ `datasets/stage4/varied_intent/StackCube-v1/samples.jsonl`, the
(failure_predicate → oracle_wrong_factor) pair distribution on
StackCube is:

| Predicate | Oracle label | Count | Rule prediction | Correct? |
|---|---|---:|---|---|
| `goal_not_satisfied` | `goal_state` | 53 | `goal_state` | ✓ |
| `direction_error`    | `goal_state` |  9 | `approach_direction` | ✗ |
| `no_motion`          | `goal_state` |  2 | `approach_direction` | ✗ |

On PushCube, **all** 44 failures are `approach_blocked → approach_direction`
and the rule is exactly right. So the head must learn:

- On StackCube, **regardless of which of {goal_not_satisfied,
  direction_error, no_motion} fires, the wrong factor is
  `goal_state`** — because the StackCube initial intent uses the
  under-specified `cube_at_target` and the only legal revision target
  is `cubeA_on_cubeB` (see Sub-project C).
- On PushCube, copy the rule.

The disambiguator is the **chosen intent's `goal_state` token**:
`cube_at_target` ⇒ under-specified ⇒ StackCube ⇒ revise `goal_state`;
anything else ⇒ the rule is already right for our two tasks.

This is a tiny separation problem — a small MLP over a handful of
one-hots and scalars should saturate it without overfitting.

## 3. Feature Vector

`vectorize_attribution_input(fp: FailurePacket, intent: Intent) → np.ndarray`
returns a fixed-dim row. **Non-privileged by construction**: only reads
FailurePacket fields (which the rule already consumes) + the agent's
own Intent (not simulator state).

| Block | Source | Dim | Notes |
|---|---|---:|---|
| failure_predicate one-hot | `fp.failure_predicate` | **9** | over `FAILURE_PREDICATES`, sorted; "none" included so the encoder works on success records too (the policy guard rejects them upstream). |
| execution_trace bools | `fp.execution_trace` | **5** | `reached_contact`, `object_moved`, `collision`, `planner_failed`, `grasp_slip` |
| object_displacement | `fp.object_displacement` | **1** | clamped to [0, 1] for stability; raw float is small but bounded by sim arena. |
| direction_alignment | `fp.direction_alignment` | **2** | `[align if not None else 0.0, is_present]` — present-flag avoids ambiguity when both endpoints zero. |
| chosen_intent factor one-hots | `intent.*` | **F bands** | one-hot over each factor's whitelist. Order: `goal_state` (4), `object_motion` (7), `contact_region` (6), `approach_direction` (5), `constraint_region` (2), `embodiment_mapping` (6) ⇒ 30. |

**Total dim** ≈ `9 + 5 + 1 + 2 + 30 = 47`. (Exact size pinned by
running the constructor in a test; see §6.)

**Firewall check (test):** the implementing module must not import
`babysteps.envs.*`, `mani_skill`, `sapien`, `torch.cuda`, or any
runtime-sim symbol. It only reads dataclass attributes. Existing
firewall tests for `stage4/features.py` are the template.

## 4. Architecture

`AttributionHead(in_dim, hidden=64, n_classes=len(INTENT_FIELDS)=6,
seed=0)` is a 2-layer MLP:

```text
Linear(in_dim, 64) → GELU → Linear(64, 6) → logits
```

- No dropout (M2a's locked simplicity bias; small data, separable).
- Deterministic init via `torch.manual_seed(seed)`.
- CPU-only at train and at eval time. No tensor cores, no AMP.

`predict(fp, intent) → str` returns argmax mapped through `INTENT_FIELDS`.

---

## 5. Training Protocol

**Data:** the union of these JSONL paths, per task:

- `datasets/stage0_baselines/{babysteps_selective,full_replan_analogue,
  one_shot,oracle_factor_revision,random_factor_revision,
  same_intent_retry,text_feedback_replan}/<task>/samples.jsonl`
- `datasets/stage4/varied_intent/<task>/samples.jsonl`

Each row contributes one training pair `(fp, intent) → oracle_wrong_factor`
when `fp.failure_predicate != "none"` and `fp.oracle_wrong_factor` is
non-null. Duplicates across baselines for the same (task, seed,
initial_intent) are **kept**: the noise model says identical features
should map to identical labels; dedup would shrink the dataset
without improving signal.

**Loss:** cross-entropy with class weights inversely proportional to
class frequency (small datasets, imbalanced classes).

**Training:** Adam, lr=1e-2, 300 epochs, batch=full-batch. Per-task
head, mirroring M2a's per-task IntentHead pattern.

**Save:** `models/stage4/m25/<task>/attribution_head.pt` +
`meta.json` recording `in_dim`, label tokens, train metadata. Loaded
into `LatentPack.attribution_head` (additive optional field).

## 6. Test Plan (TDD)

`tests/test_stage4_attribution_features.py`:

1. `test_feature_dim_is_47` — pin the dim so the model architecture
   stays stable.
2. `test_no_privileged_imports` — static string check; the module
   must not reference `mani_skill`, `sapien`, `envs.*`, `cuda`.
3. `test_predicate_one_hot_layout` — `none` is index 0; the rest
   appear in `sorted(FAILURE_PREDICATES)` order.
4. `test_intent_one_hot_layout` — factor-major order matches
   INTENT_FIELDS.
5. `test_direction_alignment_none_is_handled` — None → `[0.0, 0.0]`.

`tests/test_stage4_attribution_head.py`:

1. `test_forward_shape_is_n_by_6`.
2. `test_determinism_same_seed_same_init` — two heads with same seed
   produce bit-identical weights.
3. `test_overfits_tiny_synthetic` — 32 samples, 6 classes, linearly
   separable inputs ⇒ accuracy = 1.0 after 100 epochs (sanity).
4. `test_shuffled_labels_collapse_to_chance` — same data but shuffled
   labels ⇒ held-out acc ≤ 1/6 + 0.2 (shows the head is not
   memorizing input identity).
5. `test_predict_returns_intent_field_string` — round-trip via
   `INTENT_FIELDS`.
6. `test_save_load_roundtrip` — weights restored bit-identical.

## 7. Wire-up: `latent_revision_factory`

The existing `latent_revision_factory(pack)` builds a `RetryPolicy`
closure that:

1. Reads `ctx.demo_features`, `ctx.failure_predicate`,
   `ctx.attribution.wrong_factor`.
2. Constructs `fp_vec`, runs `ReviseHead`, decodes new factor value.
3. Emits the single-factor `Revision` with `operator="latent_revision"`.

**Change:** when `pack.attribution_head is not None`, the closure
**overrides** `ctx.attribution.wrong_factor` with the learned head's
prediction **before** step 2. The single-factor invariant is preserved
by construction: still one slot, still one factor, still one
ReviseHead call. The factor-table guard (`if factor not in
ctx.task_valid_tokens: fallback`) stays in place.

When `pack.attribution_head is None`, behavior is bit-identical to
M2a (fallback to rule). The existing M2a tests must continue to
pass without modification.

## 8. Success Criteria

1. **All TDD tests pass.**
2. **StackCube held-out attribution accuracy ≥ 0.95** on the 5-seed
   ambiguous slice (`direction_error` + `no_motion`) — sim-free
   k-fold honest acc.
3. **Shuffled-label control collapses to ≤ chance + 0.2.**
4. **Real ManiSkill StackCube G5-strict gap improves** from −10 pp to
   within −5 pp vs `oracle_factor_revision` on the existing seed-200
   to seed-249 evaluation cut.
5. **PushCube no regression**: real or sim-free smoke shows learned
   attribution ties or beats rule on PushCube.
6. **Single-factor revision invariant intact**: every emitted
   `Revision` changes exactly one factor (existing `revision.py`
   guard already enforces this; the test that asserts it must keep
   passing).

## 9. Out-of-Scope Follow-Ups

- Replacing rule-based attribution as a Stage-0 baseline. (Stage-0
  baselines are fixed; the learned head is a Stage-4 addition.)
- Cross-task shared attribution head.
- Pixel encoder (M2b, separate spec).
- Multi-candidate attribution (top-k wrong_factors): the current
  oracle is single-valued, so single-class CE is the honest fit.

---

## 10. Risks

- **Tiny training set on PushCube** (24 + 20 = 44 samples, single
  class). Risk: the head learns a uniform PushCube output and the
  intent-conditioned discrimination on StackCube is the only thing
  that matters. Mitigation: per-task heads (no PushCube/StackCube
  cross-talk) + the shuffled-label control test.
- **Direction-alignment proxies for `goal_state` correctness on
  StackCube.** This is a feature, not a bug — but the test for
  feature-only-from-FailurePacket prevents the head from
  reaching into the simulator for a richer signal.
- **Reviewer concern: "is the learned head doing anything beyond a
  task lookup?"** Answer: yes — for StackCube specifically, it
  reads three distinct predicates and emits the right factor; for
  PushCube it copies the rule. The classifier is correct exactly
  when the rule is wrong, and the diagnostic confusion-matrix
  + per-predicate breakdown in §6 makes this falsifiable.
