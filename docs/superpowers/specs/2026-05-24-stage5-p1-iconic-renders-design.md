# Stage-5 P1 — Iconic per-policy PushCube renders

## Goal

Produce 20 self-contained "full episode" MP4s — 5 held-out seeds × 4 retry
policies — that visually validate the Stage-5 P1 PushCube setup and let a
reviewer compare each policy on the same seed.

The Stage-5 P1 held-out eval (seeds 100–149, n=50) ran metrics-only; no
videos exist (`reports/stage5/p1_vision_g4_g5/PushCube-v1/report_gallery/`
shows "0/50 videos" for every policy). This work fills that gap for a
small iconic subset.

## Scope

- **Seeds (5):** `100, 110, 120, 130` (clean wins across the held-out range
  for visual variety) + `143` (the one seed where latent fails and
  oracle/babysteps succeed — diagnostic value).
- **Policies (4):** `latent` (Stage-5 vision-grounded LatentPack),
  `oracle_factor_revision`, `babysteps_selective`, `same_intent_retry`.
  Matches `reports/stage5/p1_vision_g4_g5/PushCube-v1/report_gallery/per_policy_summary.csv`
  exactly.
- **Per (seed, policy):** one continuous MP4 = demo (~3s) + initial blocked
  attempt (~6s) + that policy's retry (~6–12s). The demo and the initial
  blocked attempt frames are physically identical across the four policies
  of one seed (same env reset, same wall, same initial intent); only the
  retry differs.

Out of scope: side-by-side mosaic, other tasks (PickCube/StackCube), the
universal-failure seed 126, the rest of the 50 held-out seeds.

## Non-goals

- Not a new evaluation. Numbers come from the existing eval; this is
  documentation. No metric is recomputed.
- Not a render-pipeline refactor. We add one helper alongside the existing
  `render_baseline_contrast`; we don't change it.
- Not a reorganization of `renders/`. New output goes under a new dir
  (`renders/stage5_p1_iconic/pushcube/`); existing renders are untouched.

## Architecture

```
scripts/render_stage5_p1_iconic.py        # CLI: loops (seed × policy), writes MP4s
    │
    ├─ babysteps.envs.task_registry.get_task_entry("PushCube-v1")  # adapter + runner
    ├─ babysteps.stage4.latent_policy.load_latent_pack(             # for latent
    │       "models/stage5/p1_vision/PushCube-v1")
    ├─ load_cached_features("datasets/stage5/varied_intent/PushCube-v1/features")
    └─ for each (seed, policy):
           babysteps.render.pushcube.render_policy_episode(
               env, adapter, seed, policy_callable,
               demo_features_provider=...,
               fps=20,
           )  -> (frames_list, title_overlays)
           write_mp4(out_path, frames_list)

slurm/render_stage5_p1_iconic.sbatch       # GPU+Vulkan wrapper around the CLI
```

### `babysteps/render/pushcube.py` — new helper

```python
def render_policy_episode(
    env, adapter, seed: int, *,
    policy_name: str,                   # "latent" | "oracle_factor_revision" | …
    policy_callable,                    # RetryPolicy (RetryContext -> (Intent, Revision))
    demo_features_provider=None,        # callable(seed) -> Z, or None for non-latent
    fps: int,
) -> tuple[list, tuple]:
    """Return (concatenated_frames, (combined_title, combined_subtitle)).

    Runs phases 1 (demo), 2 (initial blocked attempt), 3 (this policy's
    retry) and returns one flat frame list. Per-phase title overlays are
    burned into the frames by `_execute_push`/`render_episode`'s existing
    annotation path so the concatenated clip still announces phase
    boundaries.
    """
```

Modeled on the existing `render_baseline_contrast` (lines 345–446). Same
wall obstacle build/move/park pattern. The latent policy needs
`demo_features_provider(seed)` to return the cached DINOv2 spatial-mean Z
that the eval used.

### `scripts/render_stage5_p1_iconic.py`

```
usage: render_stage5_p1_iconic.py
  --seeds 100,110,120,130,143
  --policies latent,oracle_factor_revision,babysteps_selective,same_intent_retry
  --pack-dir models/stage5/p1_vision/PushCube-v1
  --features-dir datasets/stage5/varied_intent/PushCube-v1/features
  --out-dir renders/stage5_p1_iconic/pushcube
  --fps 20
```

Output filename per pair:
`pushcube_seed_NNNN__<policy>_full.mp4`

20 files total.

## Tests

A sim-free smoke test in `tests/test_render_modules.py`:

- `test_render_policy_episode_smoke_fake_env` — runs the new helper with
  the fake PushCube env and a stub policy that just returns
  `same_intent_retry`'s output; asserts the returned frame list is
  non-empty and the concatenation matches `demo_len + attempt_len +
  retry_len`. Mirrors the existing `test_render_baseline_contrast` style.

No new GPU tests; the real-sim path is gated by the sbatch script.

## Risks / known unknowns

1. **Latent policy seed-coverage of cached features.** The features at
   `datasets/stage5/varied_intent/PushCube-v1/features/` must contain
   embeddings for seeds 100, 110, 120, 130, 143. If a seed is missing,
   the latent clip for that seed fails. *Mitigation:* CLI asserts feature
   presence per seed before launching env.
2. **Same-seed env reset reproducibility across the four policy passes.**
   The render helper resets the env per policy pass, so the initial cube
   pose must be identical for all four policies of one seed (verified by
   the existing `_pushcube_setup` which seeds from `seed`).
3. **Vulkan availability on the assigned compute node.** Standard project
   risk; the existing sbatch scripts in `slurm/` demonstrate the working
   pattern.
